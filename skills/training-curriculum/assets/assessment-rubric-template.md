## Writing Good Anchor Descriptions

Each anchor description should be:

- **Specific to the criterion.** "Good work" is not an anchor. "Handles all edge cases including null inputs, empty collections, and concurrent access" is.
- **Observable.** Describe what the evaluator can see, not what the learner "understands."
- **Differentiated across levels.** Each level must be clearly distinguishable from the adjacent levels. If you cannot tell the difference between a 3 and a 4, the anchors need rewriting.
- **Calibrated to the audience.** A rubric for new hires has different expectations than one for senior engineers.

### Example Criterion: Error Handling (for a coding assessment)

| Score | Anchor Description |
|-------|-------------------|
| 5 | Handles all expected error cases with specific, informative error messages. Includes edge cases (null, empty, concurrent). Errors are logged with context. Recovery mechanisms are implemented where applicable. |
| 4 | Handles all common error cases with clear error messages. Most edge cases covered. Errors are logged. |
| 3 | Handles primary error cases (network failure, invalid input). Error messages are present but may be generic. Some edge cases missed. |
| 2 | Handles 1-2 error cases. Error messages are vague or missing. Multiple common failure modes unhandled. |
| 1 | No error handling. Application crashes on invalid input or network failure. |
