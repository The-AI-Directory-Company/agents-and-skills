## Governance Notes

### Who owns this table?
_[Role or team responsible for maintaining and approving changes — e.g., "Content Design team" or "Product + Design jointly."]_

### How to propose a new term
1. Check this table to confirm the concept is not already covered.
2. Propose the new term in _[channel/tool — e.g., "a PR to this file" or "#ux-copy Slack channel"]_.
3. Include: the preferred term, rejected alternatives, and a one-sentence rationale.
4. The table owner reviews and approves or suggests a revision.

### When to update
- **New feature launches** — Any new user-facing concept needs a row before UI copy is written.
- **User research findings** — If research shows users do not understand a term, propose a replacement.
- **Terminology conflicts** — If two teams use different words for the same thing, resolve it here.

### Enforcement
- **UI copy**: All interface text must use preferred terms. Code review should catch violations.
- **Documentation**: Docs team follows this table. Linting rules can flag rejected terms in markdown.
- **Marketing**: Landing pages and emails should align. Minor variation is acceptable for SEO purposes only if documented here.
- **Support**: Help articles and macros use preferred terms. Retrain search synonyms so rejected terms still surface the right articles.

### Review cadence
Audit the table quarterly. For each term, ask:
- Is it still used consistently across all surfaces?
- Has user feedback suggested the term is confusing?
- Have any new rejected alternatives appeared in the wild?
